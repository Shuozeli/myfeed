# myfeed

Personal feed bot that crawls social media via pwright browser automation
and sends updates to Telegram.

## Quick Orientation

```
src/
  main.rs        CLI entry point: run (daemon), once (single cycle), login (manual auth), events (debug)
  config.rs      All settings from env vars. Panics on any missing var (fail-fast, no defaults).
  crawler.rs     Loads pwright recipe YAML, runs via pwright_script::executor::execute(),
                 parses output into CrawledItem {external_id, title, url, preview, raw}.
  scheduler.rs   Async loop: for each enabled site, connect Chrome -> run recipe -> dedup -> telegram.
                 Uses spawn_blocking for sync diesel calls.
  db.rs          SQLite via diesel. Mutex<SqliteConnection>. All queries in transactions.
                 Tables: feed_items (dedup by site+external_id), event_log (time-series debug).
  schema.rs      Auto-generated by diesel CLI. Do not hand-edit.
  telegram.rs    HTTP client for Telegram Bot sendMessage API. Formats items as HTML.

recipes/
  explore/       HTML structure discovery scripts -- run these FIRST before writing feed recipes.
                 Output raw DOM info (selectors, counts, sample HTML) for a site.
  <site>-feed.yaml  Feed extraction recipes. Each must output a JSON array via the `items` key:
                     [{id, title, url, preview}, ...]

migrations/      Diesel SQL migrations. Run via embedded migrations at startup (db.migrate()).
docs/            architecture.md, design.md, codelabs.md, tasks.md
```

## How the Crawl Pipeline Works

1. `scheduler.rs` timer fires (default 30min)
2. For each site in `ENABLED_SITES`:
   - Opens a new Chrome tab via `Browser::connect()` + `browser.new_tab("about:blank")`
   - Loads `recipes/<site>-feed.yaml` via `pwright_script::parser::parse_yaml_file()`
   - Executes recipe against the tab's Page via `pwright_script::executor::execute()`
   - Recipe navigates to the site, waits for content, runs JS eval to extract posts
   - `crawler.rs` parses the recipe's output (looks for `items` key with JSON array)
   - Each item is checked against `feed_items` table (unique on `site + external_id`)
   - New items: inserted into DB + sent to Telegram via `TelegramBot::send_feed_item()`
   - Tab is closed
3. Events logged to `event_log` table: `crawl_start`, `crawl_complete`, `crawl_error`

## Key Dependencies

- `pwright-bridge` -- Chrome CDP connection, Browser, TabHandle, Page
- `pwright-script` -- Recipe parser, executor, VecSink for capturing output
- `pwright-cdp` -- Low-level CDP types (imported transitively)
- `diesel` -- SQLite ORM with migrations
- `reqwest` -- HTTP client for Telegram API
- `tokio` -- Async runtime
- `clap` -- CLI argument parsing

All pwright crates are git dependencies from `https://github.com/shuozeli/pwright.git`.

## Build & Test

```bash
cargo build                           # Build
cargo build --release                 # Release build
./target/release/myfeed once          # Test single crawl (needs Chrome + .env)
./target/release/myfeed events -l 10  # Check event log
```

No unit tests yet -- the project depends on live Chrome for meaningful testing.
Integration testing requires Chrome with remote debugging + valid .env config.

## Rules

### Architecture
- **Browser-only crawling.** Do NOT add API-based crawlers. Everything goes through
  pwright recipes. This is a design principle, not a temporary limitation.
- **Recipe-based extraction.** Site-specific DOM logic lives in YAML recipes, not Rust.
  Adding a new site should require zero Rust code changes (except adding its login URL
  to the `Login` command match in `main.rs`).
- **Stateless tabs.** Each crawl opens a fresh tab and closes it when done.
  Do NOT pool tabs or reuse sessions across crawls.

### Code
- All DB queries MUST be wrapped in diesel transactions (`conn.transaction(|conn| {...})`)
- Config values MUST NOT have defaults -- panic on missing env vars
- Do NOT commit `.env` or `*.db` files
- Do NOT hand-edit `src/schema.rs` -- it's generated by `diesel print-schema`
- Recipes must output items as JSON array with keys: `id`, `title`, `url`, `preview`

### Design Docs
- Design docs go in `docs/designs/` with filename prefix `YYMMDD-` (e.g., `260323-atom-feed-output.md`)
- Write the design doc BEFORE implementing. Get alignment first.

### Git
- Do NOT commit or push unless explicitly asked
- Do NOT amend commits unless explicitly asked

## Adding a New Site

1. Create `recipes/explore/<site>-explore.yaml` (copy an existing one, change selectors)
2. Run it via pwright CLI to understand the DOM: `pwright script run recipes/explore/<site>-explore.yaml`
3. Create `recipes/<site>-feed.yaml` with JS that extracts `[{id, title, url, preview}]`
4. Add the site name to `ENABLED_SITES` in `.env`
5. Add login URL to `main.rs` `Command::Login` match arm

## Common Modifications

| Task | Where to change |
|------|----------------|
| Adjust crawl interval | `CRAWL_INTERVAL_SECS` in `.env` |
| Change Telegram message format | `telegram.rs` `send_feed_item()` |
| Add a new site | See "Adding a New Site" above |
| Fix broken selectors | Edit the site's `recipes/<site>-feed.yaml` |
| Add a new CLI command | `main.rs` `Command` enum + match arm |
| Change DB schema | New migration in `migrations/`, then `diesel print-schema > src/schema.rs` |
| Add keyword filtering | `crawler.rs` `parse_outputs()` or new filter step in `scheduler.rs` |
